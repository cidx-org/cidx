# CIDX — Product Guardrails

## Why this document exists

CIDX was built to solve a real problem: reduce friction between local work, validation, and automated execution, without forcing projects into a rigid framework.

This document protects CIDX against its main threat: becoming, over time, a framework more concerned with its own model than with the real needs of teams.

The goal is simple: evolve CIDX without betraying its founding philosophy.

## Core principle

CIDX must adapt to the project.

The project must never be forced to bend to fit CIDX.

---

## The 5 non-negotiable rules

### 1. CIDX is an engine, not a doctrine

CIDX is not a universal methodology.
CIDX is not a delivery religion.
CIDX is not a global governance system.

CIDX must remain a coherent, readable, and useful execution engine.

If an evolution turns CIDX into a prescriptive framework, that evolution must be refused.

### 2. No phase should exist without real value

A step should only exist if it addresses a concrete, experienced need.

No phase should be introduced:
- to complete a theoretical schema
- to appear "mature"
- to reflect an ideal abstract cycle
- to satisfy a pipeline aesthetic

If a phase doesn't bring clear operational value, it must not exist.

### 3. Simplicity is a design constraint

Simplicity is not a lack of ambition.
It's a requirement.

Every new feature must answer a simple question:
- does it remove a real friction?
- or does it add a layer of sophistication?

If the answer is ambiguous, the feature must be delayed or refused.

### 4. The user must always stay in control

CIDX must help, not take over.

System behavior must remain:
- understandable
- predictable
- observable
- overridable

The user must be able to:
- see what will be executed
- understand why
- limit execution to what's necessary
- override a behavior when justified

### 5. Abstraction must stay at the right level

CIDX can abstract execution, context, useful orchestration.

CIDX must not over-abstract:
- team organization
- governance
- a single vision of delivery
- a complete process imposed on everyone

The higher abstraction climbs toward doctrine, the further CIDX strays from its mission.

---

## Anti-goals

CIDX must not become:

- a complete organizational transformation framework
- a collection of mandatory phases
- a sprawling configuration layer
- a product centered on itself
- a machine whose maintenance mainly serves to sustain its own complexity
- a tool that requires ideological buy-in to be useful

---

## Feature acceptance criteria

Every new feature should satisfy the majority of these criteria:

- it addresses a real pain point from the field
- it simplifies more than it complicates
- it respects the "tool adapts to the project" principle
- it doesn't force a single workflow
- it remains understandable without excessive documentation
- it can be introduced without making the product core more opaque
- it brings a concrete gain locally, in automation, or in their continuity

If not, it probably shouldn't be built.

---

## Drift signals

If several of these signals appear, CIDX is starting to drift:

- features are added without identified real usage
- configuration grows faster than value produced
- the product starts imposing its structure on projects
- phases or concepts exist without a clear need
- it becomes harder to simply explain what CIDX does
- documentation talks more about the model than the problem solved
- the mental cost of adoption increases
- the product starts demanding ideological adherence
- real exceptions are treated as deviations to correct rather than needs to respect

---

## Questions to ask before any major evolution

Before adding a capability:

- what real problem does this solve?
- how many times has this problem been encountered?
- is this a field friction or an architecture desire?
- does this evolution simplify the work or just the model?
- does this make CIDX more useful, or just more complete?
- if this feature didn't exist, would the product be worse for teams, or just less impressive?

---

## Decision rule

If an idea:
- increases CIDX's control over projects
- significantly increases complexity
- imposes a cycle or structure
- or demands doctrinal adherence

then the default decision must be: **no**.

The burden of proof must remain on the side of complexity, not on the side of simplicity.

---

## What CIDX must protect above all

- continuity between local work and automated execution
- behavior readability
- adaptability to real contexts
- product core sobriety
- user confidence in what will be executed

---

## What DevOps means here

DevOps is not about tooling for the sake of tooling.

DevOps, here, means:
- reducing silos
- bringing responsibilities closer
- making the flow more continuous
- keeping quality and security close to the action
- preventing each team from working in its own corridor
- building systems that serve collaboration instead of ritualizing it

CIDX does not exist to stage an ideal cycle.
CIDX exists to help teams better execute reality.
