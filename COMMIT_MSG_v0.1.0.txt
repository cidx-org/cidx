feat: initial release - convention-based CI/CD orchestration

CIDX v0.1.0 introduces a declarative, convention-over-configuration approach to
CI/CD pipeline orchestration using Docker containers and preset tool configurations.

## Core Features

### Convention Over Configuration
- **Preset-based tool execution**: 20+ built-in presets (trivy, gitleaks, prettier,
  golangci-lint, go-test, godog, gh-release, docker-buildx, commitizen, etc.)
- **Zero-boilerplate pipelines**: Named pipelines with automatic phase execution
- **Smart defaults**: Tools run with sensible configurations out of the box

### Environment-Aware Execution
- **Automatic environment detection**: Local vs CI (GitHub Actions, GitLab CI, Jenkins, CircleCI)
- **Local safety modes**: Semi dry-run behavior for dangerous operations (release, docker)
  - `draft` mode: GitHub releases created as drafts locally
  - `no-push` mode: Docker images built without registry push
  - Automatic activation without flags - no `--dry-run` needed
- **CI production mode**: Full execution with deployment capabilities

### Event-Driven Pipelines
- **Context-aware orchestration**: Different Git events trigger appropriate phases
  - Pull Request → validation only (security, code, test)
  - Merge to main → validation + build (security, code, test, build)
  - Tag push → complete pipeline (security, code, test, build, release, docker)
- **Named pipelines**: `pr`, `main`, `release`, `publish`, `quick`, `pre-push`

### Behavior-Driven Development
- **Gherkin specifications**: 7 feature files defining CIDX behavior
  - Event handling (PR, tag, merge, direct commit)
  - Local safety modes and environment detection
  - Pipeline execution and phase orchestration
- **Living documentation**: BDD scenarios serve as executable specifications
- **Dogfooding**: CIDX tests itself using godog and its own pipelines

## Architecture

### Phase-Based Organization
- **security**: Vulnerability scanning (trivy, gitleaks)
- **code**: Quality checks (prettier, golangci-lint, commitizen)
- **test**: Test execution (go-test, godog)
- **build**: Artifact creation (go-build)
- **release**: Artifact publication (gh-release, goreleaser)
- **docker**: Container image publication (docker-buildx, kaniko)

### Docker-Native Execution
- **Container-based isolation**: Each tool runs in its own Docker container
- **Volume mounting**: Workspace and configuration files automatically mounted
- **Environment expansion**: Variables resolved at runtime (${WORKSPACE}, ${TAG})
- **Container reuse**: Cache preservation for tools like trivy

### DevOps Loop Alignment
Maps CI/CD phases to DevOps stages:
- **CODE** → security, code phases (validation & quality)
- **BUILD** → test, build phases (artifact creation)
- **RELEASE** → release, docker phases (artifact publication)
- Out of scope: PLAN (project management), OPERATE (continuous), MONITOR (observability)

## Philosophy

### DOCKER = RELEASE (Not Deploy)
- **Release phase**: Publishes artifacts to stores (GitHub Releases, GHCR, Docker Hub)
- **Deploy phase** (future): Runs applications in production (kubectl, docker-compose, cloud-run)
- **Key distinction**: `docker push` publishes artifacts, does NOT deploy infrastructure

### Convention Over Complexity
- **No verbose YAML**: Define phases and tools, CIDX handles execution
- **Smart event detection**: No conditions needed - CIDX knows what to run
- **Minimal configuration**: ~20 lines vs 200+ in traditional CI/CD configs

## Configuration

### Minimal Setup
```toml
[security]
tools = ["trivy", "gitleaks"]

[code]
tools = ["prettier", "golangci-lint", "commitizen"]

[test]
tools = ["go-test", "godog"]

[build]
tools = ["go-build"]

[pipelines.release]
phases = ["security", "code", "test", "build", "release", "docker"]
```

### Available Pipelines
- `ci`: Full CI validation (security, code, test, build)
- `pr`: Pull request validation (security, code, test)
- `main`: Main branch pipeline (security, code, test, build)
- `release`: Complete release pipeline (all phases)
- `publish`: Artifact publication only (release, docker)
- `quick`: Fast code quality check (code only)
- `pre-push`: Local validation before push (security, code, test)

## Usage

```bash
# Local development (safe mode - automatic)
cidx run pr           # Validate before creating PR
cidx run pre-push     # Validate before pushing
cidx run release      # See what release would do (semi dry-run)
cidx run publish      # See artifact publication (semi dry-run)

# CI environment (production mode - automatic)
cidx run release      # Full release with GitHub + GHCR publication
```
